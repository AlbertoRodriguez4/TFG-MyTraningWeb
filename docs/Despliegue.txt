MANUAL COMPLETO DE DESPLIEGUE - THE TRAINING HUB
=================================================

1. ARQUITECTURA GENERAL
-----------------------
El proyecto esta dividido en tres capas desplegadas sobre Amazon Web Services (AWS):

- Frontend: Aplicacion Vue 3 compilada y servida mediante Nginx dentro de un contenedor Docker.
- Backend: API REST desarrollada en .NET 8 (C#), tambien dentro de un contenedor Docker.
- Base de Datos: PostgreSQL. Se recomienda usar Amazon RDS en produccion para mayor fiabilidad.

La orquestacion de los contenedores se realiza con Kubernetes a traves de Amazon EKS.
Cada componente expone sus servicios mediante un LoadBalancer de AWS.

3. DOCKERFILES
--------------

A) Frontend (Frontend/Dockerfile)
   - Etapa 1: Usa node:lts-alpine para compilar la aplicacion Vue.
     * Instala dependencias con npm install.
     * Compila con npm run build.
   - Etapa 2: Usa nginx:stable-alpine para servir los archivos estaticos.
     * Copia el resultado de la compilacion a /usr/share/nginx/html.
     * Expone el puerto 80.

B) Backend (Backend/Dockerfile)
   - Etapa 1: Usa mcr.microsoft.com/dotnet/sdk:8.0 para compilar.
     * Restaura paquetes con dotnet restore.
     * Publica en modo Release con dotnet publish.
   - Etapa 2: Usa mcr.microsoft.com/dotnet/aspnet:8.0 para ejecutar.
     * Copia los binarios publicados.
     * Expone el puerto 6873.
     * Punto de entrada: dotnet aa2-alberto-rodriguez-penalva.dll

4. MANIFIESTOS DE KUBERNETES (EKS)
----------------------------------

A) front-deployment.yml
   - Deployment: 1 replica del contenedor Nginx.
   - Service: Tipo LoadBalancer, puerto 80.
   - Imagen utilizada: albertorodriguezpenalva/front-aws:latest

B) api-deployment.yml
   - Deployment: 1 replica del contenedor .NET 8.
   - Service: Tipo LoadBalancer, puerto 6873.
   - Imagen utilizada: albertorodriguezpenalva/back-aws:latest
   - Variables de entorno:
     * ASPNETCORE_ENVIRONMENT=Development
     * ConnectionStrings__DefaultConnection: cadena de conexion a PostgreSQL.
       (En produccion se recomienda externalizar este dato a AWS Secrets Manager.)

C) postgres-deployment.yml (solo local/dev)
   - Deployment con la imagen oficial de PostgreSQL.
   - Variables: POSTGRES_PASSWORD, POSTGRES_USER, POSTGRES_DB.
   - Service interno en el cluster, puerto 5432.

D) pgadmin-deployment.yml (solo local/dev)
   - Deployment con la imagen dpage/pgadmin4.
   - Variables: PGADMIN_DEFAULT_EMAIL, PGADMIN_DEFAULT_PASSWORD.
   - Service tipo LoadBalancer, puerto 80.

5. CI/CD CON GITHUB ACTIONS
---------------------------
Existen dos pipelines independientes para Frontend y Backend.

A) docker-image-cicd-front.yml
   - Disparadores: push o pull_request sobre main que modifiquen archivos en Frontend/**
   - Pasos:
     1. Checkout del codigo.
     2. Configurar Docker Buildx.
     3. Login en Docker Hub.
     4. Generar metadata (tags/labels).
     5. Build y push de la imagen:
        - Contexto: ./Frontend
        - Dockerfile: ./Frontend/Dockerfile
        - Tag: latest
        - Plataforma: linux/amd64
        - Cache habilitada (GitHub Actions).
     6. Configurar credenciales AWS.
     7. Actualizar kubeconfig para conectar con EKS.
     8. Aplicar manifiesto: kubectl apply -f Frontend/manifest/front-deployment.yml
     9. Forzar reinicio del deployment: kubectl rollout restart deployment front

B) docker-image-cicd-back.yml
   - Disparadores: push o pull_request sobre main que modifiquen archivos en Backend/**
   - Pasos:
     1. Checkout del codigo.
     2. Configurar Docker Buildx.
     3. Login en Docker Hub.
     4. Generar metadata (tags/labels).
     5. Build y push de la imagen:
        - Contexto: ./Backend
        - Dockerfile: ./Backend/Dockerfile
        - Tag: latest
        - Plataforma: linux/amd64
        - Cache habilitada.
     6. Configurar credenciales AWS.
     7. Actualizar kubeconfig para conectar con EKS.
     8. Aplicar manifiesto: kubectl apply -f Backend/manifest/api-deployment.yml
     9. Forzar reinicio del deployment: kubectl rollout restart deployment api

6. SECRETS NECESARIOS EN GITHUB
--------------------------------
Para que los pipelines funcionen, se deben configurar los siguientes secrets
en el repositorio (Settings > Secrets and variables > Actions):

- DOCKER_USERNAME: Usuario de Docker Hub.
- DOCKER_PASSWORD: Contrasena o token de acceso de Docker Hub.
- AWS_ACCESS_KEY_ID: Access Key de AWS IAM.
- AWS_SECRET_ACCESS_KEY: Secret Key correspondiente.
- AWS_SESSION_TOKEN: Token de sesion (obligatorio para cuentas de AWS Academy/estudiantes).

7. PRERREQUISITOS PREVIOS AL DESPLIEGUE
----------------------------------------

- Cuenta de AWS activa.
- Cluster EKS creado (ejemplo: mi-cluster en us-east-1).
- AWS CLI instalado y configurado localmente:
    aws configure
- kubectl instalado y configurado para el cluster:
    aws eks update-kubeconfig --name mi-cluster --region us-east-1
- Imagenes Docker subidas a Docker Hub (o a Amazon ECR).
- Base de datos PostgreSQL disponible (preferiblemente Amazon RDS).

8. DESPLIEGUE MANUAL (sin CI/CD)
--------------------------------

A) Frontend
    cd Frontend
    docker build -t albertorodriguezpenalva/front-aws:latest .
    docker push albertorodriguezpenalva/front-aws:latest
    kubectl apply -f manifest/front-deployment.yml
    kubectl rollout restart deployment front

B) Backend
    cd Backend
    docker build -t albertorodriguezpenalva/back-aws:latest .
    docker push albertorodriguezpenalva/back-aws:latest
    kubectl apply -f manifest/api-deployment.yml
    kubectl rollout restart deployment api

9. COMANDOS UTILES PARA VERIFICAR EL ESTADO
--------------------------------------------

Ver todos los pods:
    kubectl get pods

Ver todos los servicios y sus IPs publicas:
    kubectl get svc

Ver logs del backend:
    kubectl logs -l app=api --tail=100

Ver logs del frontend:
    kubectl logs -l app=front --tail=100

Ver descripcion detallada de un pod:
    kubectl describe pod <nombre-del-pod>

Escalar replicas manualmente:
    kubectl scale deployment front --replicas=3
    kubectl scale deployment api --replicas=3

10. CONEXION ENTRE FRONTEND Y BACKEND
-------------------------------------
El frontend debe apuntar a la URL publica del backend.
Esto se configura en las variables de entorno del proyecto Vue, por ejemplo:

    VITE_API_URL=http://<EXTERNAL-IP-DEL-BACKEND>:6873

Si se utiliza un Ingress o dominio propio, se puede enrutar /api al backend
y el resto al frontend sin necesidad de exponer puertos independientes.

11. LIMPIEZA DE RECURSOS
------------------------
Si se desea eliminar los despliegues del cluster:

    kubectl delete -f Frontend/manifest/front-deployment.yml
    kubectl delete -f Backend/manifest/api-deployment.yml

12. CHECKLIST PARA ENTORNO DE PRODUCCION
-----------------------------------------

- Cambiar ASPNETCORE_ENVIRONMENT de Development a Production.
- Usar Amazon RDS en lugar del contenedor local de PostgreSQL.
- No incluir contrasenas directamente en los manifiestos de Kubernetes.
  Utilizar AWS Secrets Manager o Kubernetes Secrets.
- Configurar un dominio personalizado (Route 53).
- Habilitar SSL/TLS mediante AWS Certificate Manager.
- Implementar un Ingress Controller para gestionar el trafico HTTP/HTTPS.
- Habilitar monitoreo con CloudWatch, Prometheus o Grafana.
- Revisar y ajustar las politicas de IAM aplicando el principio de minimo privilegio.

13. NOTAS IMPORTANTES
---------------------

- El workflow original proporcionado apuntaba a rutas como ./aa2-Alberto/.
  Los actuales workflows ya estan adaptados a la estructura real del proyecto:
  Frontend/ y Backend/.

- El despliegue de postgres y pgadmin existente en Backend/manifest/
  esta pensado unicamente para pruebas locales. En AWS se recomienda
  desactivarlos o eliminarlos y usar Amazon RDS.

- El force rollout restart en el pipeline garantiza que Kubernetes descargue
  la ultima imagen con el tag latest, ya que por defecto no lo hace si la
  etiqueta no cambia.
