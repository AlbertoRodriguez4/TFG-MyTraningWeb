 He encontrado varios patrones de código duplicado e inconsistencias similares al caso consistencyStreak:                                     
                            
  🔍 Hallazgos de Código Duplicado/Inconsistente                                                                                               
  
  1. iscompleted vs isCompleted ⚠️ CRÍTICO                                                                                                        
  Problema: El modelo Routine.ts usa isCompleted pero toda la aplicación usa iscompleted.                                                      
  
  // src/components/Models/Routine.ts:8
  isCompleted: boolean;  // ← Definición incorrecta

  // src/components/Models/Routines.ts:7
  iscompleted: boolean;  // ← Correcto (usado en toda la app)

  Impacto: 20+ archivos usan iscompleted, pero el modelo define isCompleted. Esto rompe la seguridad de tipos.

  2. createdat vs createdAt ℹ️ MENOR

  Todo el frontend usa createdat (snake_case del backend). Es consistente pero no sigue camelCase de TypeScript. No es crítico porque viene del
   backend así.

  3. userid vs userId ⚠️ IMPORTANTE

  Problema: Hay inconsistencia entre stores y modelos:

  // Models (correcto - camelCase)
  src/components/Models/Purchase.ts: userId: number
  src/components/Models/PurchasedItem.ts: userId: number
  src/components/Models/Plan.ts: userId: number

  // Stores (incorrecto - mezcla)
  src/stores/UsersRoomStore.ts: userid: number (línea 8)
  src/stores/PurchaseStore.ts: addPurchase(userid: number, ...) (línea 34)
  src/stores/RoomStore.ts: userid: number (línea 66)

  4. passwordhash ℹ️ CONSISTENTE

  Todo el código usa passwordhash (snake_case). Es consistente en todos lados, aunque no sea camelCase ideal. Viene así del backend.

  5. equippedStrengthItemId / equippedEnduranceItemId ✅ CONSISTENTE

  Estos campos están correctamente unificados en camelCase en todo el frontend.
